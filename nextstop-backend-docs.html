Interswitch Payment Flow

Secure server-to-server verification — never trust the client

1
Flutter calls POST /api/payments/initiate
Flutter sends rideId and payerType (INITIATOR or JOINER). The server computes the correct amount using the fare-splitting algorithm.
{ "rideId": "uuid", "payerType": "INITIATOR" }
2
Server generates txRef + SHA512 Hash
A unique txRef (NSP-timestamp-uuid) is generated. A SHA512 hash is computed using ProductID + PayItemID + txRef + Amount + RedirectURL + ClientSecret. Credentials never leave the server.
hash = SHA512(ProductID + PayItemID + txRef + amountKobo + redirectURL + ClientSecret)
3
PENDING transaction record inserted to MySQL
Before redirecting the user, we persist the transaction in MySQL with status=PENDING. This enables idempotency and audit trail.
INSERT INTO transactions (tx_ref, amount_kobo, status='PENDING', ...)
4
Flutter opens Webpay URL in WebView
The server returns a paymentUrl. Flutter renders this in a WebView. The user completes payment on Interswitch's hosted page.
{ "paymentUrl": "https://sandbox.interswitchng.com/collections/w/pay?..." }
!
User returns to app — DO NOT trust Flutter's status
After Webpay redirect, Flutter reports a status. This is UNTRUSTED. A malicious client could fake a success. Flutter must call /api/payments/verify next.
5
Flutter calls POST /api/payments/verify
Flutter sends the txRef. The server initiates a server-to-server GET to Interswitch — completely independent of the client's reported status.
GET https://sandbox.interswitchng.com/collections/api/v1/gettransaction.json ?productid=PRODUCT_ID&transactionreference=txRef&amount=amountKobo Authorization: Bearer {access_token} Hash: {SHA512(ProductID + txRef + ClientSecret)}
✓
Dual validation: ResponseCode + Amount Match
The server validates TWO conditions simultaneously. Both must pass or the transaction is rejected.
iswData.ResponseCode === '00' // ✓ Interswitch confirms success Number(iswData.Amount) === storedTx.amount_kobo // ✓ No amount tampering
6
MySQL updated + ride status advanced
Only after both validations pass: transaction status set to SUCCESS, ride status advanced to ACCEPTED, full ISW response stored as JSON for audit.
UPDATE transactions SET status='SUCCESS', isw_raw_response='{...}' UPDATE rides SET status='ACCEPTED'
7
Async: Webhook at POST /webhooks/interswitch
Interswitch also POSTs asynchronous events (SUCCESS, REVERSAL). The webhook validates HMAC-SHA512 signature, checks idempotency via webhook_events table, then re-verifies the transaction server-side.
// Signature validation (constant-time comparison — prevents timing attacks) crypto.timingSafeEqual( Buffer.from(req.headers['x-interswitch-signature'], 'hex'), Buffer.from(HMAC-SHA512(rawBody, WEBHOOK_SECRET), 'hex') )

Response 201
"accessToken": "eyJhbGci...", // 15min expiry "refreshToken": "eyJhbGci..." // 30d expiry

Response includes user object + tokens
"user": { "id": "uuid", "fullName": "Seyi Adebayo", "role": "PASSENGER", "ninVerified": false, "isVerified": true }
POST /auth/verify-nin Driver NIN verification via ISW KYC 🔒 JWT + DRIVER ▼
Request Body
"nin": "12345678901" // 11 digits
What happens server-side
// 1. Fetch OAuth2 token from ISW Passport (cached) // 2. GET ISW KYC API with NIN // 3. On responseCode='00': update users.nin_verified=true // 4. Return {verified: true} — NIN never echoed to client

🚌 Rides

POST /rides Initiator creates a new ride 🔒 JWT + PASSENGER ▼
Request Body
"pickupAddress": "14 Admiralty Way, Lekki Phase 1", "pickupLat": 6.4301, "pickupLng": 3.4533, "dropoffAddress": "Victoria Island, Lagos", "dropoffLat": 6.4281, "dropoffLng": 3.4219, "rideType": "RIDESHARE", // or "SOLO" "maxJoiners": 2, "baseFare": 3000, // NGN "womenOnly": false
Response 201
"rideId": "uuid", "status": "REQUESTED", "fareBreakdown": { "baseFare": 3000, "driverEarnings": 2550, // 85% "platformFee": 450, // 15% "initiatorPays": 3000 // 0 joiners yet }
GET /rides/available?lat=6.43&lng=3.45&radius=5 Joiner discovers nearby RIDESHARE rides 🔒 JWT ▼
Returns — sorted by distance (Haversine)
[{ "id": "uuid", "pickupAddress": "Lekki Phase 1", "dropoffAddress": "Victoria Island", "baseFare": 3000, "currentJoiners": 1, "maxJoiners": 2, "initiatorName": "Seyi Adebayo", "jobTitle": "Senior Engineer", "distance_km": 1.23 }]
PATCH /rides/:rideId/status Advance ride state machine 🔒 JWT ▼
Valid Transitions
REQUESTED → NEGOTIATING | ACCEPTED | CANCELLED NEGOTIATING → ACCEPTED | CANCELLED ACCEPTED → ARRIVED | CANCELLED ARRIVED → IN_PROGRESS IN_PROGRESS → COMPLETED
Request Body
"status": "ARRIVED", "cancelReason": "Traffic too heavy" // optional
Real-time Side Effect
// Socket.io event emitted to ride room io.to(`ride:${rideId}`).emit('ride:status', { rideId, status })
POST /rides/:rideId/join Joiner requests to join a rideshare 🔒 JWT + PASSENGER ▼
Response 201 — split fare pre-calculated
"status": "PENDING", "estimatedSplitFare": 1000 // ₦3000 / 3 passengers
Initiator is notified via Socket.io
io.to(`user:${initiatorId}`).emit('ride:join_request', { rideId, userId, splitFare: 1000 })

Fare Engine

Interactive cost-sharing simulator — Equal Split Model

Algorithm Logic

totalPassengers = acceptedJoiners + 1 // Initiator counts splitFareEach = baseFare / totalPassengers // Equal split platformFee = baseFare × 0.15 // 15% of base driverEarnings = baseFare - platformFee // 85% always initiatorPays = splitFareEach // Reduced by joiners joinerPays = splitFareEach // Same rate // Driver always earns 85% of baseFare regardless of joiners // Joiners save money; platform earns its cut from all passengers
